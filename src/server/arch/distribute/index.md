# 分布式原理

分布式系统是多台计算机协同工作的系统，目标是提高性能、可用性、可扩展性。理解分布式系统的原理和挑战，是构建大规模系统的基础。

## CAP 定理

### CAP 的定义

CAP 定理指出，分布式系统无法同时满足一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）这三个目标，最多只能同时满足两个。

一致性：所有节点在同一时间看到相同的数据。可用性：每个请求都能得到响应（成功或失败）。分区容错性：系统在网络分区（节点间无法通信）时仍能继续运行。

### CAP 的权衡

CA：放弃分区容错性，意味着不能容忍网络分区，适用于单机系统。CP：放弃可用性，当网络分区发生时选择暂停服务以保证数据一致，适用于金融等对一致性要求高的场景。AP：放弃强一致性，接受数据可能不一致，但保证服务始终可用，适用于社交网络等场景。

大多数分布式系统选择 AP，因为网络分区不可避免，暂停服务是不可接受的。AP 系统通过最终一致性保证数据最终一致。

### BASE 理论

BASE 理论是对 AP 系统的指导。Basically Available（基本可用）：系统即使出现局部故障，依然能响应请求。Soft state（软状态）：数据状态允许有一段时间的滞后。Eventual consistency（最终一致性）：数据在更新后，可能在几秒钟内不同节点看到的数值不一致，但经过一段时间，所有节点最终会达到相同状态。

## 分布式一致性

### 强一致性

强一致性要求所有节点在同一时间看到相同的数据。实现方式：两阶段提交（2PC）、三阶段提交（3PC）、Paxos、Raft。强一致性的代价：性能差（需要同步）、可用性差（需要多数节点可用）。

### 最终一致性

最终一致性允许数据短暂不一致，但保证最终一致。实现方式：Gossip 协议、版本向量、CRDT（Conflict-free Replicated Data Types）。最终一致性的好处：性能好、可用性好。最终一致性的代价：应用需要处理不一致数据（如显示"同步中"）。

### 一致性级别

读后一致性：客户端读取自己写入的数据，保证能看到最新的写。单调读：客户端读取的数据版本不会回退。会话一致性：会话内的读后一致性。因果一致性：有因果关系的操作，所有节点看到的顺序一致。

## 分布式共识

### Paxos

Paxos 是 Leslie Lamport 提出的分布式共识算法，用于在多个节点间达成一致。Paxos 的角色：Proposer（提议者）、Acceptor（接受者）、Learner（学习者）。Paxos 的过程：Prepare（准备阶段，Proposer 请求 Acceptor 承诺不再接受旧提议）、Accept（接受阶段，Proposer 提交提议）、Learn（学习阶段，Learner 获取已接受的提议）。

Paxos 的问题：理解困难、实现复杂。Multi-Paxos 是 Paxos 的优化，通过选举 Leader 减少协商次数。

### Raft

Raft 是 Diego Ongaro 提出的分布式共识算法，目标是易于理解。Raft 的角色：Leader（领导者，处理所有请求）、Follower（跟随者，响应 Leader 请求）、Candidate（候选人，选举时的临时角色）。

Raft 的过程：Leader 选举（任期（Term）机制，每个任期选举一次 Leader）、日志复制（Leader 接收请求，追加到日志，复制到 Follower，多数节点确认后提交）、安全性（日志连续性、Leader 完整性）。

Raft 的优势：易于理解、实现简单、广泛使用（etcd、Consul、TiKV）。

### ZAB

ZAB（ZooKeeper Atomic Broadcast）是 ZooKeeper 的原子广播协议，用于保证 ZooKeeper 集群的一致性。ZAB 的过程：Leader 选举（epoch 机制，类似 Raft 的 Term）、消息广播（Leader 接收消息，广播给 Follower，多数节点确认后提交）、崩溃恢复（Leader 崩溃后，新 Leader 恢复未提交的消息）。

## 分布式事务

### 2PC（两阶段提交）

2PC 是分布式事务的协议，保证多个参与者原子提交。2PC 的过程：准备阶段（Prepare，协调者询问参与者是否可以提交）、提交阶段（Commit，如果所有参与者都同意，则提交，否则回滚）。

2PC 的问题：同步阻塞（参与者等待协调者指令，无法处理其他请求）、单点故障（协调者故障，参与者无法释放资源）、数据不一致（协调者发送提交消息后崩溃，部分参与者未收到提交）。

### 3PC（三阶段提交）

3PC 是 2PC 的改进，增加 CanCommit 阶段，减少阻塞时间。3PC 的过程：CanCommit（协调者询问参与者是否可以提交）、PreCommit（如果所有参与者都同意，则预提交）、DoCommit（协调者发送提交指令）。

3PC 的问题：仍然阻塞、数据不一致风险仍然存在。

### TCC（Try-Confirm-Cancel）

TCC 是应用层的分布式事务协议，将业务逻辑分成三个阶段：Try（尝试执行，预留资源）、Confirm（确认执行，使用预留资源）、Cancel（取消执行，释放预留资源）。

TCC 的好处：性能好（无锁）、灵活（可以自定义补偿逻辑）。TCC 的代价：代码侵入（需要实现三个接口）、幂等性（Confirm/Cancel 可能重复调用）。

### Saga

Saga 是长事务的解决方案，将长事务拆分成多个本地事务，每个本地事务有补偿事务。Saga 的执行：依次执行本地事务，如果某个事务失败，则执行已执行事务的补偿事务。

Saga 的问题：补偿事务可能失败（需要重试）、脏读（未提交的数据被其他事务读取）、隔离性差（需要应用层控制）。

### 本地消息表

本地消息表是将消息存储在本地数据库，在同一事务中写入业务数据和消息，然后异步发送消息。消息发送失败时，定时任务重试。

本地消息表的好处：可靠性高（消息和业务在同一事务）、实现简单。本地消息表的代价：需要定时任务、可能重复发送（需要幂等性）。

## 分布式锁

### 分布式锁的实现

基于 Redis：SET key value NX EX seconds（设置 key，如果不存在则成功，过期时间防止死锁）。基于 ZooKeeper：创建临时顺序节点，最小的节点获得锁。基于数据库：SELECT FOR UPDATE（悲观锁）、唯一索引（乐观锁）。

### 分布式锁的问题

死锁：持有锁的进程崩溃，锁无法释放。解决：设置过期时间、心跳续约。单点故障：Redis 或 ZooKeeper 崩溃，锁服务不可用。解决：集群部署。

性能：分布式锁的性能受限于网络延迟和锁服务的性能。解决：分段锁（减少锁竞争）、乐观锁（减少锁使用）。

分布式系统是复杂的，CAP 定理、分布式共识、分布式事务、分布式锁都是分布式系统的基础问题。理解这些问题的原理和权衡，有助于设计合适的分布式架构。
