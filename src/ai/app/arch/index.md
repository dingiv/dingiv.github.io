# 应用架构
把 LLM 集成到应用中只是第一步，让它可靠、高效、成本可控地运行在生产环境中是另一回事。一个能跑的 demo 和一个可运营的生产系统之间，隔着模型路由、API 网关、语义缓存、级联降级等一系列架构模式。这些模式解决的核心问题是：如何在模型能力、成本和延迟之间找到动态平衡。

## 架构演进
LLM 应用的架构经历了三代演进。第一代是单模型直连——前端直接调用 OpenAI API，简单但脆弱，模型故障则应用瘫痪。第二代是网关代理——通过 API 网关（LiteLLM、OpenRouter）统一管理多模型调用，支持负载均衡和故障切换。第三代是智能路由——根据请求特征动态选择模型和策略，简单问题用小模型（快且便宜），复杂问题用大模型（慢但准确），相似问题直接返回缓存结果。

这种演进与传统 Web 应用的架构演进类似：单机 → 反向代理 → 微服务 + 服务网格。核心驱动力都是可用性、性能和成本。LLM 应用的特殊性在于"模型即服务"的非确定性——同样的输入可能产生不同的输出，模型的响应时间和质量也在不断变化，架构需要应对这种动态性。

## 架构层次
![arch](./arch.dio.svg)


一个成熟的 LLM 应用架构通常包含以下层次：

**客户端层**处理用户输入、流式渲染、状态管理。这一层在 `frontend.md` 中有详细讨论。

**API 网关层**统一入口，负责认证鉴权、限流计费、请求路由。网关对外暴露 OpenAI 兼容协议，对内代理到不同模型提供商。这一层使得应用代码不需要关心底层模型的变化。

**路由层**根据请求内容选择最合适的模型。路由策略可以是基于规则的（关键词匹配 → 小模型，复杂推理 → 大模型），也可以是基于模型的（用小模型判断任务复杂度，再路由到对应模型）。

**缓存层**拦截重复或相似的请求，直接返回缓存结果。语义缓存通过向量相似度匹配"意思相近"的问题，比精确匹配的缓存命中率高得多。

**模型服务层**执行实际的 LLM 推理。可以是自建的 vLLM/Ollama 服务，也可以是云端 API（OpenAI、Anthropic、Google）。每个模型有独立的健康检查和性能指标。

**可观测层**贯穿所有层次，记录每次请求的完整链路（输入→路由→模型→输出），用于质量评估、成本追踪和故障排查。这一层在 [LLMOps](/ai/app/ops/) 中详细讨论。

## 关键设计决策
在构建 LLM 应用架构时，几个关键决策需要根据业务场景权衡：

**同步 vs 异步**：简单问答用同步调用即可，用户等待几秒获得回答。长文档处理、批量分析等任务需要异步模式——立即返回任务 ID，用户通过轮询或回调获取结果。异步模式需要任务队列（Redis Queue、Celery）和状态管理，复杂度显著增加。

**有状态 vs 无状态**：多轮对话需要维护会话状态（消息历史、用户偏好），这引入了状态管理的复杂性。无状态设计（每次请求携带完整上下文）更简单但成本更高（重复传输历史）。通常的做法是热数据在 Redis，冷数据在数据库，按需加载。

**中心化 vs 边缘化**：中心化的模型服务便于管理和优化，但增加了网络延迟。边缘化部署（在每个区域运行推理实例）降低延迟但增加运维成本。对于实时性要求高的场景（语音 Agent），边缘化是必要的。
