

## 1. 交互范式的变革：从 GUI 到 LUI
传统的图形用户界面（GUI）是基于按钮和菜单的确定性交互，而 AI 时代推崇 **LUI (Language User Interface)**。
* **流式渲染 (Streaming)**：学习如何处理 `Server-Sent Events (SSE)`。AI 回答不是一次性给出的，前端需要像打字机一样实时渲染 Markdown 或代码块。
* **生成式 UI (Generative UI)**：这是目前最前沿的方向（如 Vercel 的 AI SDK）。根据 AI 的返回结果，动态渲染不同的 React 组件（比如 AI 说“这是今天的股市”，前端自动弹出一个 Recharts 图表组件，而不仅仅是文本）。



---

## 2. 前端 AI 开发全家桶
你不需要从底层写起，但必须精通这些专门为 AI 设计的前端框架：
* **Vercel AI SDK**：目前前端集成 AI 的事实标准，支持 React, Next.js, Vue, Svelte。它完美处理了流式传输、聊天状态管理（useChat）和工具调用（Tool Calling）的 UI 状态。
* **LangChain.js**：如果你想在浏览器端直接运行一些轻量级的逻辑（如本地向量化或简单的链式调用），这是必学的。
* **Model-as-a-Service (MaaS) 调用**：熟悉如何在前端安全地调用 OpenAI、Anthropic 或 Hugging Face 的 API（通常通过环境变量和后端 Proxy）。

---

## 3. 结构化数据的处理与解析
AI 返回的是字符串，但你的组件需要对象。
* **JSON 约束与校验**：学习使用 **Zod** 或 **TypeChat**。在发送请求时定义好 Schema，强制要求 AI 返回符合你前端组件要求的 JSON 格式，避免渲染崩溃。
* **Markdown & Latex 渲染**：AI 经常返回复杂的公式和代码，你需要精通 `react-markdown`、`prismjs` 和 `katex` 的集成。

---

## 4. 浏览器端 AI (Web AI)
随着硬件能力的提升，越来越多的 AI 任务可以直接在前端运行：
* **Transformers.js**：在浏览器里直接跑模型。比如实现本地的图片分类、文本翻译或语音转文字（Whisper），无需经过后端。
* **WebGPU**：了解如何利用浏览器的 GPU 加速能力。2026 年，很多轻量级 LLM（如 Gemma 或 Llama-3-Tiny）已经可以直接在 Chrome 浏览器里跑，这对减少后端成本极其重要。



---

## 5. 提示词工程的前端化 (Prompt in Frontend)
前端开发者需要设计更聪明的“输入增强”：
* **Context Injection**：在用户发送消息前，前端自动收集当前页面状态、选中的文本或用户偏好，并作为隐形 Context 拼接到 Prompt 中。
* **提示词模板预览**：设计让用户能直观感受到 Prompt 调整对结果影响的 UI 调试工具。

---

## 6. 状态管理的新挑战
* **对话树管理**：AI 对话不再是线性的，用户可能会“重新生成”、“分支对话”。你需要设计复杂的状态树来管理这些历史记录。
* **多模态输入处理**：熟练处理图片上传预览、语音录入动画、以及文件切片上传给 RAG 系统。

---

# 一、岗位本质（先统一认知）

LLM Application Developer ≠ 算法工程师
本质是：

> **用大模型做“系统工程”，而不是做模型**

你要解决的是：

* 模型不稳定（hallucination）
* 延迟高 / 成本高
* 数据脏 / 检索不准
* 系统不可控

---

## 2️⃣ RAG 系统（你总结的核心，但再补一层）

你已经写得很好，但要补一个关键：

👉 **RAG 本质是 IR（信息检索系统）+ LLM 的组合**

面试官真正看的是：

### （1）召回质量优化（Recall）

* Dense（embedding）
* Sparse（BM25）
* Hybrid（必须会）

👉 不会 hybrid，直接降级

---

### （2）精排（Ranking）

* Cross-encoder rerank
* 多阶段 pipeline：

```text
召回（top 50） → 精排（top 5） → prompt
```

---

### （3）上下文构造（Context Engineering）

这是很多人不会的：

* 去重 / 去冲突
* 时间优先级（最新数据优先）
* 多文档融合（不是简单拼接）

---

### （4）抗幻觉设计（重点）

企业最关心这个：

* “只允许引用 context”
* 引用来源（citation）
* fallback（不知道就说不知道）

---

## 3️⃣ 向量数据库（不是 CRUD，是系统设计）

你写了 HNSW / IVF，但面试更关心：

### （1）索引选型 trade-off

| 索引   | 特点      |
| ---- | ------- |
| HNSW | 高精度，内存大 |
| IVF  | 可扩展，需调参 |
| PQ   | 压缩，精度下降 |

👉 面试会问：

> 1 亿数据怎么选？

---

### （2）工程问题（更重要）

* 冷启动（索引构建时间）
* 更新成本（增量 vs 重建）
* 内存 vs 磁盘（热数据）

---

### （3）真实瓶颈

不是查询慢，而是：

```text
embedding 生成慢 + rerank 慢
```

---

# 四、面试官最爱问的问题（直接给你重点）

1. **RAG 不准怎么办？**

   * chunk → recall → rerank → prompt 全链路分析

2. **如何降低 hallucination？**

   * 限制上下文 + 引用机制 + fallback

3. **向量数据库选型？**

   * 数据量 + 延迟 + 更新频率

4. **Agent 为什么会死循环？怎么解决？**

   * max step + 状态检查 + reward / termination

5. **如何优化成本？**

   * caching + routing + prompt 压缩
