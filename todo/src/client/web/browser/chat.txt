## 1. 性能指标量化与度量工具（DevTools & Lighthouse）

面试官最反感“我觉得变快了”这种主观描述，他们要看的是你对核心 Web 指标（Core Web Vitals）的绝对掌控。

### 核心考点：

* **指标深挖**：Lighthouse 中的 FP、FCP、**LCP（最大内容渲染）**、**INP（交互延迟，React 19/2026年核心指标）**、**CLS（累积布局偏移）** 的底层物理意义是什么？如何精准优化它们？
* **工具微操**：你在 Chrome DevTools 的 **Performance 面板**里，如何通过火焰图（Flame Chart）定位长任务（Long Tasks）？**Memory 面板**里的三种快照（Heap Snapshot、Allocation Instrumentation）分别在什么场景下使用？

### 💡 答题要点（大厂级标准）：

* **LCP 优化**：LCP 通常是首屏的大图或大文本。要说出如何通过 `<link rel="preload" as="image">` 提升 LCP 元素的下载优先级，并结合 `fetchpriority="high"` 确保它不被其他静态资源阻塞。
* **CLS 优化**：CLS 衡量的是页面的稳定性。要指出在项目初期，必须为所有异步加载的图片、广告位、画布节点**显式声明宽高（Width/Height）或使用 `aspect-ratio` 占位**，防止内容加载出来后产生剧烈的“视觉跳跃”。
* **INP / FID 优化**：INP（Interaction to Next Paint）在 2026 年已经全面取代 FID 成为核心指标。要阐述如何通过 Performance 面板抓取超过 **50ms 的长任务**，并通过 `requestIdleCallback`、`MessageChannel` 或将大任务拆分为微任务/宏任务切片，释放主线程，从而把 INP 控制在 200ms 的优秀线内。

## 2. 首屏加载优化（First Screen Optimization）

### 核心考点：

* 完整的资源加载与传输调优：网络层（HTTP 协议特征落地）+ 构建层（产物控制）。

### 💡 答题要点（体系化输出）：

面试官让你谈谈首屏优化时，不要零散地蹦出几个词，要按照**网络传输 $\rightarrow$ 资源体积 $\rightarrow$ 执行时机**的管线流转来回答：

1. **网络与传输层**：
* 开启 **HTTP/2 或 HTTP/3 的多路复用**，消除浏览器的同源并发连接限制。
* 配置合理的**缓存策略**：第三方基础底座（React/Vue/Canvas引擎）使用 `Cache-Control: max-age=31536000, immutable` 强缓存；业务频繁变动的 index.html 使用 `no-cache` 配合 `ETag` 协商缓存。
* 利用 **CDN 边缘节点**加速静态资源的就近分发。


2. **构建体积层**：
* 结合上一项技能，阐述通过 Webpack/Vite 实施 `SplitChunks`，将核心依赖与路由页面进行 **Code Splitting（代码分割）**，首屏只下载当前路由必需的 Bundle。
* 使用 Modern Mode 针对现代浏览器直接打包 Native ESM，减少庞大的 ES5 Polyfill 注入。


3. **关键渲染路径（CRP）**：
* 实施 **SSR / RSC（结合前文的服务端组件）**，直接在服务端输出结构化 VNode 或 HTML，将传统的“白屏等 JS 下载并渲染”变成“首屏即视”。
* 对非核心 JS 脚本加上 `defer` 或 `async`，防止其阻塞浏览器的 HTML 解析（DOM 构建）。

## 3. 长列表虚拟滚动（Virtual Scrolling）

### 核心考点：

* 虚拟滚动的底层数学模型与高频高负载场景下的性能防线。

### 💡 答题要点（硬核算法解析）：

* **核心原理**：只渲染当前可视区域（Viewport）内的 DOM 节点，配合上下少量的缓冲区（Buffer）。通过计算滚动条的 `scrollTop`，动态切片数组（`slice(startIndex, endIndex)`），并利用 `transform: translateY()` 或绝对定位将可视节点撑起到正确的物理位置。
* **面试官追问：如果列表项的高度是动态不固定的（如朋友圈、带有不定长评论的卡片），你怎么设计虚拟滚动？**
* *完美答卷*：采用“预估高度 + 动态修正”策略。


1. 初始化时，为每条数据设定一个预估高度（Estimated Height），计算出大致的总高度并撑开滚动条。
2. 在节点真正渲染入 DOM 后，利用 **`ResizeObserver`** 或在 `updated` 周期中通过 `getBoundingClientRect()` 抓取其**真实物理高度**。
3. 动态更新内部的**位置索引缓存表（Offset Map）**，重新修正滚动条的总高以及后续所有节点的 `translateY` 偏移量。为了防止高频滚动导致 ResizeObserver 密集触发引发掉帧，必须配合 `requestAnimationFrame` 进行节流操作。

## 4. 图片懒加载（Lazy Loading）

### 核心考点：

* 现代浏览器原生的懒加载技术与老旧方案的演进对比。

### 💡 答题要点：

* **老旧方案的缺陷**：监听 `scroll` 事件，利用 `getBoundingClientRect().top <= window.innerHeight` 判断图片是否进入视口。*痛点*：`scroll` 事件触发极频，且调用 `getBoundingClientRect` 会**强行触发浏览器的同步布局重算（Forced Synchronous Layout）**，极易引发卡顿。
* **现代最优解**：
1. 使用 **`IntersectionObserver`（交叉观察器）**。它是异步的、基于浏览器底层发布订阅机制的 API，只有当图片元素与视口发生交叉时才会触发回调，性能极高。
2. 直接利用现代浏览器原生的 **`loading="lazy"`** 属性。在 `<img src="..." loading="lazy" />` 上声明后，浏览器会自动推迟视口外图片的下载，无需写一行 JS。

## 5. 内存泄漏排查（Memory Leak Investigation）

这是区分开发人员是“真有实战经验”还是“纯背八股”的分水岭，面试官会让你复述一次真实的排查链路。

### 核心考点：

* JS 垃圾回收机制（V8 的标记清除与分代回收）+ 真实排查手段。

### 💡 答题要点（场景化答法）：

* **高频内存泄漏场景**：
1. **未销毁的定时器或事件监听**：在组件销毁（Unmount）时，忘记清除 `setInterval` 或 `window.addEventListener`，导致整个组件上下文被闭包死死引用，无法被 GC 回收。
2. **游离的 DOM 节点（Detached DOM）**：在 JS 中保存了某个 DOM 节点的引用，随后通过 `removeChild` 从页面删除了该 DOM。由于 JS 变量还攥着这个引用，导致该 DOM 节点虽然从屏幕上消失了，但依然静静躺在内存中。
3. **全局变量或未清理的闭包（如项目二中的全局 Agent 会话上下文）**。


* **教科书级排查链路复述**：
> “在实际项目中，当发现页面越用越卡、或者在 Performance 面板中看到 **JS Heap（内存折线图）呈现阶梯式上升、在组件销毁后从不下降时**，我会启动内存排查三板斧：
> 1. 打开 Chrome DevTools 的 **Memory 面板**，选择 **Heap Snapshot（堆快照）**。
> 2. 首先在页面初始化时拍下 **快照 1**；接着反复进行疑似泄漏的操作（如反复打开关闭弹窗 10 次），拍下 **快照 2**；最后强制触发一次浏览器的垃圾回收（点击垃圾桶图标），拍下 **快照 3**。
> 3. 利用 Memory 面板的 **Comparison（对比）** 功能，让快照 3 与快照 1 进行比对。
> 4. 在结果中筛选出 **`Detached HTMLDivElement`（游离DOM）** 或特定的组件构造函数名。通过底层的 **Retainers（持有者引用链）** 向上追溯，能够一眼看清究竟是哪一个全局变量、闭包或未清理的全局事件监听器锁死了这片内存。定位到代码行后，在组件销毁生命周期中对其进行精准释放。”
